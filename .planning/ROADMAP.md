## 단계 상세

### 1단계: 프로젝트 설정

**목표:** 개발을 위한 모노레po 인프라와 데이터베이스 스키마가 준비된다

**의존:** 없음 (첫 단계)

**요구사항:** BACK-04

**성공 기준** (사용자 관점에서 무엇이 참인가):
1. 개발자가 `pnpm install`로 모든 의존성을 설치할 수 있다
2. 개발자가 `pnpm --filter backend dev`와 `pnpm --filter frontend dev`로 개발 서버를 실행할 수 있다
3. Prisma 마이그레이션으로 SQLite 데이터베이스가 생성된다
4. Biome으로 코드 린트/포맷팅이 실행된다

### 2단계: 인증 시스템

**목표:** 사용자가 계정을 생성하고 브라우저 세션 전체에서 로그인 상태를 유지할 수 있다

**의존:** 1단계 (Prisma User 모델, 백엔드 인프라)

**요구사항:** AUTH-01, AUTH-02, AUTH-03, SEC-02

**성공 기준** (사용자 관점에서 무엇이 참인가):
1. 사용자가 이메일과 비밀번호로 회원가입할 수 있다
2. 사용자가 로그인 후 브라우저를 새로고침해도 로그인 상태가 유지된다
3. 사용자가 로그아웃하면 세션이 종료되고 로그인 페이지로 리다이렉트된다
4. JWT 토큰이 만료 5분 전에 자동 갱신되어 긴 대화 중 세션이 끊기지 않는다

### 3단계: 채팅 핵심

**목표:** 사용자가 OpenRouter 무료 모델과 실시간 스트리밍으로 대화하고 관리할 수 있다

**의존:** 2단계 (인증된 사용자 컨텍스트)

**요구사항:** CHAT-01, CHAT-02, CHAT-03, CHAT-04, CONV-01, CONV-02, CONV-03, CONV-04, BACK-01, BACK-02, BACK-03, BACK-05, BACK-06

**성공 기준** (사용자 관점에서 무엇이 참인가):
1. 사용자가 메시지를 전송하면 토큰 단위로 스트리밍 응답이 실시간으로 표시된다
2. 사용자가 "Stop generating"으로 AI 응답 생성을 중단할 수 있다
3. 사용자가 "Regenerate"로 마지막 AI 응답을 재생성할 수 있다
4. 사용자가 새 대화를 생성하고 사이드바에서 최근 순으로 목록을 조회할 수 있다
5. 사용자가 대화를 삭제할 때 확인 다이얼로그가 표시된다
6. 대화 제목이 첫 메시지 내용을 기반으로 자동 생성된다
7. 사용자가 대화별 시스템 프롬프트를 설정할 수 있다
8. API rate limit(429) 발생 시 사용자 친화적 메시지와 재시도 버튼이 표시된다

### 4단계: UI/UX 구현

**목표:** 사용자가 반응형 인터페이스에서 마크다운 콘텐츠를 보고 다크 모드를 전환할 수 있다

**의존:** 3단계 (채팅 데이터 및 기능)

**요구사항:** DISP-01, DISP-02, DISP-03, DISP-04, UIUX-01, UIUX-02, UIUX-03, UIUX-04, UIUX-05, UIUX-06

**성공 기준** (사용자 관점에서 무엇이 참인가):
1. AI 응답이 마크다운으로 렌더링된다 (코드 블록, 표, 목록 포함)
2. 사용자가 코드 블록의 복사 버튼으로 코드를 한 번에 복사할 수 있다
3. 사용자가 AI 전체 응답을 복사할 수 있다
4. 사용자가 다크 모드와 라이트 모드를 전환할 수 있다 (다크 모드 기본)
5. 사용자가 모바일과 데스크톱에서 모든 기능을 사용할 수 있다
6. 사용자가 사이드바를 접고 펼 수 있다
7. AI 응답 생성 중 로딩 인디케이터가 표시된다
8. 새 대화 빈 화면에 온보딩 안내와 추천 프롬프트가 표시된다
9. 에러 발생 시 사용자 친화적 메시지와 재시도 버튼이 표시된다
10. 멀티라인 입력창이 내용에 따라 높이가 자동 조절된다
11. 스트리밍 중 자동 스크롤되고, 사용자가 스크롤하면 일시 정지된 후 하단 이동 버튼이 표시된다

### 5단계: 보안 및 안정화

**목표:** API 키가 안전하게 보호되고 모든 입력이 검증되며 예외가 우아하게 처리된다

**의존:** 3단계 (모든 API 엔드포인트)

**요구사항:** SEC-01

**성공 기준** (사용자 관점에서 무엇이 참인가):
1. API 키가 환경변수로 관리되며 클라이언트(브라우저 네트워크 탭 포함)에 절대 노출되지 않는다
2. 모든 API 요청이 DTO 기반으로 검증되어 잘못된 입력은 400 에러로 반환된다
3. 백엔드 에러가 사용자 친화적 메시지로 변환되어 프론트엔드에 표시된다
4. OpenRouter API 에러가 적절히 로깅되고 사용자에게 명확하게 전달된다
