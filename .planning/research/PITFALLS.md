# 도메인 위험 요소 연구

**도메인:** AI 채팅 웹 애플리케이션 (OpenRouter 통합)
**조사일:** 2026-03-29
**신뢰도:** MEDIUM

## 치명적 위험 요소

### 위험 1: 스트리밍 연결 누수

**문제 상황:**
사용자가 페이지를 이동하거나 새로고침할 때 스트리밍 연결이 정리되지 않아 서버 리소스가 계속 점유됨. NestJS 백엔드에서 OpenRouter API로의 연결이 열린 상태로 유지되어 메모리 누수 발생.

**발생 원인:**
- 클라이언트 연결 해제 시 서버에서 스트림 정리 로직 미구현
- `AbortController`로 HTTP 요청 취소하지 않음
- 데이터베이스 트랜잭션이 스트림 중단 시 롤백되지 않음

**예방 방법:**
- NestJS에서 `OnGatewayDisconnect` 또는 요청 취소 리스너로 스트림 정리
- 클라이언트에서 `AbortSignal`을 사용하여 요청 취소 전파
- 항상 `try...finally`로 스트림 리소스 정리 보장
- 데이터베이스 작업을 스트림 완료 후 커밋

**경고 신호:**
- 개발자 도구 Network 탭에서 "pending" 상태 요청이 누적
- 서버 메모리 사용량이 시간 경과에 따라 지속적 증가
- 데이터베이스 연결 풀이 고갈됨

**대응 단계:** Phase 1 (인프라 및 API 설계)

### 위험 2: 스트리밍 응답 중단 시 메시지 불일치

**문제 상황:**
사용자가 "생성 중단"을 클릭하거나 네트워크 오류 발생 시, 이미 전송된 토큰들이 메시지로 저장되지 않거나, 완료되지 않은 메시지가 UI에 표시됨. 사용자가 "재시도" 시 이전 불완전 응답과 중복 생성됨.

**발생 원인:**
- 스트리밍 중단을 정상적인 완료 상태와 구분하지 않음
- 부분 응답을 저장할 로직이 없음
- "재시도" 시 마지막 사용자 메시지를 다시 전송하여 중복 생성

**예방 방법:**
- 메시지 상태 플래그 도입: `streaming`, `completed`, `stopped`, `error`
- 중단 시 수신된 토큰들을 "중단된 메시지"로 저장
- 재시도 시 완료된 마지막 어시스턴트 메시지를 기준으로 재생성 요청
- UI에서 중단된 메시지에 "중단됨" 표시 추가

**경고 신호:**
- 채팅 기록에서 누락된 응답이 존재
- 재시도 시 이전 응답이 채팅창에서 사라짐
- 데이터베이스 메시지 테이블에서 status 컬럼 누락

**대응 단계:** Phase 2 (채팅 핵심 기능)

### 위험 3: OpenRouter Rate Limiting 예외 누락

**문제 상황:**
OpenRouter API가 rate limit에 도달했을 때 429 응답을 반환하지만, 클라이언트에 적절한 오류 메시지를 전달하지 않음. 사용자는 "알 수 없는 오류"만 보고 재시도를 반복하여 상황 악화.

**발생 원인:**
- OpenRouter의 에러 응답 형식을 파싱하지 않음
- HTTP 429를 일반 서버 오류로 처리
- Retry-After 헤더를 무시하고 즉시 재시도
- 사용자에게 rate limit 상황을 명확히 전달하지 않음

**예방 방법:**
- OpenRouter 에러 응답 구조 정의 및 타입 안전한 파싱
- HTTP 429 발생 시 Retry-After 헤더 확인 후 지연 재시도
- 사용자에게 "요청이 너무 많습니다. 잠시 후 다시 시도해주세요" 명시적 메시지
- 프론트엔드에서 요청 속도 제한 (throttle/debounce)

**경고 신호:**
- 개발자 도구 콘솔에서 429 에러 로그만 존재하고 사용자 피드백 없음
- 재시도 버튼 클릭 시 즉시 다시 실패
- OpenRouter API 키가 일시적 사용 정지됨

**대응 단계:** Phase 2 (API 통합)

### 위험 4: JWT 토큰 만료 후 스트리밍 중단

**문제 상황:**
긴 스트리밍 응답 중 JWT 토큰이 만료되면, 다음 요청부터 401 Unauthorized로 실패. 사용자는 로그아웃 처리되지만 진행 중이던 스트림은 계속되어 혼란.

**발생 원인:**
- 스트리밍 시작 시에만 토큰 검증
- 토큰 갱신(refresh token) 로직 미구현
- 401 발생 시 사용자에게 명확한 안내 없이 로그아웃

**예방 방법:**
- Access 토큰 만료 5분 전에 자동 갱신
- 401 발생 시 refresh 토큰으로 재발급 시도
- 갱신 실패 시 사용자에게 "세션 만료. 다시 로그인해주세요" 안내
- 진행 중인 스트림은 완료시키고, 새 요청부터 인증 요구

**경고 신호:**
- 긴 응답 생성 중 갑자기 로그아웃됨
- 401 에러 후 사용자가 무엇을 해야 할지 모름
- 토큰 만료 시간이 하드코딩되어 있음

**대응 단계:** Phase 1 (인증 시스템)

### 위험 5: 모노레포 의존성 호이스팅 충돌

**문제 상황:**
pnpm workspace에서 frontend와 backend가 서로 다른 버전의 공통 의존성(예: zod, typescript)을 참조하여 타입 불일치 발생. 빌드는 성공하지만 런타임에 타입 에러 또는 직렬화 오류 발생.

**발생 원인:**
- `pnpm-workspace.yaml`에서 패키지 간 공유 의존성 명시하지 않음
- 각 패키지의 `package.json`에서 버전을 고정하여 버전 드리프트
- 공유 타입 정의를 위한 별도 패키지 미존재
- TypeScript `paths` 설정이 실제 node_modules와 불일치

**예방 방법:**
- 루트 `package.json`에 공통 의존성 선언 및 워크스페이스에서 상속
- 공유 타입을 위한 `packages/shared` 패키지 생성
- 모든 타입 정의를 shared 패키지에 배치
- `pnpm.overrides`로 특정 패키지 버전 강제 통일
- CI에서 의존성 일관성 체크 (`pnpm dedupe --check`)

**경고 신호:**
- `import type` 시 타입 에러가 없지만 런타임에 속성 불일치
- frontend/backend에서 `node_modules`에 설치된 버전이 다름
- 공유 인터페이스 변경 시 한쪽에서만 에러 발생

**대응 단계:** Phase 0 (프로젝트 설정)

## 기술적 부채 패턴

당장은 편해 보이지만 장기적으로 문제가 되는 지름길.

| 지름길 | 당장 이득 | 장기적 비용 | 허용 조건 |
|--------|-----------|-------------|-----------|
| 프론트엔드에서 OpenRouter API 직접 호출 | 백엔드 프록시 없이 빠른 prototyping | API 키 노출 보안 위험, rate limiting 불가 | 절대 불가 |
| 스트리밍 상태를 Zustand만으로 관리 | 간단한 상태 로직 | 서버 상태와 불일치, 새로고칭 시 소실 | prototyping 단계만 |
| 하드코딩된 모델 allowlist | DB 스키마 없이 빠른 시작 | 모델 변경 시 재배포 필요, 동적 관리 불가 | MVP까지 허용, 그 이후 개선 |
| SQLite 사용 지속 | 설정 간소 | 동시성 제약, 프로덕션 부적합 | 100명 미만 사용자까지 |
| 에러 메시지 영어 유지 | i18n 없이 빠른 개발 | 한국어 사용자 경험 저하 | 절대 불가 (한국어 서비스) |

## 통합 고착 사항

외부 서비스 연결 시 자주 발생하는 실수.

| 서비스 | 일반적인 실수 | 올바른 접근법 |
|--------|---------------|---------------|
| OpenRouter API | 에러 응답 본문을 읽지 않고 HTTP 상태코드만 확인 | `error: { message, code }` 구조 파싱하여 사용자에게 표시 |
| NestJS SSE | `@Sse()` 데코레이터만 사용하고 Observable 완료 처리 안 함 | `complete()` 또는 `error()`로 명시적 스트림 종료 |
| Prisma SQLite | 개발 시 SQLite 사용 후 프로덕션에 그대로 배포 | PostgreSQL로 마이그레이션 계획 수립 및 환경 분리 |
| JWT 인증 | Access 토큰 만료 기간을 30일로 설정 | 15분 만료 + Refresh 토큰 조합 |
| TanStack Query | `staleTime: 0`으로 모든 요청을 항상 refetch | 채팅 목록은 5분, 사용자 정보는 30분 등 상황에 맞게 설정 |

## 성능 함정

소규모에서는 동작하지만 규모 시 실패하는 패턴.

| 함정 | 증상 | 예방 | 실패 규모 |
|------|------|------|-----------|
| 클라이언트에서 무제한 메시지 렌더링 | 스크롤 시 점점 느려짐 | 가상 스크롤 또는 메시지 페이지네이션 | 100+ 메시지 |
| 스트리밍 토큰마다 React 리렌더링 | CPU 사용량 100%, 브라우저 멈춤 | 토큰 배치 처리 (100ms마다 한 번씩) | 긴 응답 생성 시 |
| 채팅 목록 전체를 항상 refetch | 페이지 로딩 2초 이상 지연 | TanStack Query 캐싱 + invalidate 전략 | 10+ 대화 |
| 중단된 스트림 정리 없음 | 메모리 사용량 지속 증가 | `AbortController`로 요청 취소 및 리소스 정리 | 하루 사용 후 |
| markdown 렌더링 최적화 없음 | 코드 블록 포함 메시지 렌더링 1초+ | React.memo, markdown-it 직렬화 캐싱 | 긴 코드 응답 시 |

## "완성된 것처럼 보이지만 아닌 것" 체크리스트

완성된 것처럼 보이지만 핵심이 누락된 것들.

- [ ] **스트리밍 인터럽션:** 중단 기능이 있지만 진행 중인 스트림이 실제로 취소되지 않음 — 네트워크 탭에서 pending 상태 확인
- [ ] **에러 복구:** API 오류 발생 시 "다시 시도" 버튼이 있지만 같은 요청을 반복하여 다시 실패 — 실패 원인 해결 후 재시도되는지 확인
- [ ] **메시지 상태:** 메시지가 표시되지만 DB에 저장되지 않아 새로고침 시 소실 — 데이터베이스 조회 후 일치하는지 확인
- [ ] **토큰 만료:** 로그아웃 기능이 있지만 토큰 만료 후 자동 갱신 안 됨 — 15분 대기 후 재요청으로 확인
- [ ] **타입 안전성:** TypeScript에 에러가 없지만 런타임에 API 응답 형식 불일치 — 실제 API 응답으로 테스트
- [ ] **Rate Limiting:** UI에는 속도 제한 메시지가 있지만 실제로는 제한 없음 — 빠르게 연속 요청으로 확인

## 회복 전략

예방에도 불구하고 위험이 발생했을 때의 복구 방법.

| 위험 | 복구 비용 | 복구 단계 |
|------|-----------|-----------|
| 스트리밍 연결 누수 | HIGH | 1. 서버 재시작으로 누수 정리<br>2. 모든 스트림에 AbortController 추가<br>3. 연결 모니터링 로직 추가 |
| 메시지 불일치 | MEDIUM | 1. 데이터베이스에서 완료되지 않은 메시지 찾기<br>2. status 컬럼 추가 마이그레이션<br>3. 중단 로직 개선 |
| Rate Limiting 미작동 | LOW | 1. OpenRouter 대시보드에서 사용량 확인<br>2. 에러 파싱 로직 추가<br>3. 사용자 안내 메시지 개선 |
| JWT 만료 로직 미구현 | MEDIUM | 1. 현재 토큰 만료 기간 확인<br>2. refresh 토큰 발급 로직 추가<br>3. 401 핸들러에서 자동 갱신 |
| 의존성 충돌 | MEDIUM | 1. `pnpm ls`로 버전 차이 확인<br>2. 공통 의존성을 루트로 이동<br>3. `pnpm dedupe` 실행 |

## 위험 요소-단계 매핑

로드맵 단계에서 이 위험 요소들을 어떻게 다루어야 하는지.
